# Security policy

개인정보 분류, 역할별 권한 행렬, AI 처리 흐름과 보존·파기 기준은 [PRIVACY_ACCESS_DESIGN.md](./PRIVACY_ACCESS_DESIGN.md)를 기준으로 합니다.

## Reporting a vulnerability

보안 문제에는 개인정보, 인증 우회, Firestore 규칙 오류, 노출된 자격 증명과 공급망 문제가 포함됩니다. 공개 이슈에 민감한 재현 정보나 비밀값을 게시하지 말고 저장소 관리자에게 비공개 채널로 전달해 주세요.

## Deployment checklist

- 실제 운영에서는 Firebase Authentication과 Firestore 동기화를 함께 검증한 후 활성화합니다.
- 실제 사용자 운영 배포는 `VITE_DEMO_MODE_ENABLED=false`를 강제합니다. 현재 실제 사용자가 없는 공개 데모 기간에는 Auth와 Firestore 동기화를 함께 끄고 가상 데이터를 브라우저에만 저장합니다.
- Firebase 웹 API 키는 클라이언트 식별자이지만 Google Cloud에서 허용 도메인과 API를 제한하고, App Check 및 예산 알림을 설정합니다.
- 서비스 계정 키, 개인 키, 토큰과 비밀번호는 `VITE_` 변수나 Git 이력에 저장하지 않습니다.
- `firestore.rules`와 인덱스를 애플리케이션 배포 전에 별도로 검증하고 배포합니다.
- 상담 원문 메모는 `consultationNotes`에 분리하고 학생 계정에 읽기 권한을 부여하지 않습니다. 학생은 `studentVisible=true`인 요약만 조회합니다.
- 상담사는 `counselorUid`가 본인인 담당 학생·상담·원문 메모·예약만 조회합니다. 사용자 승인과 학생 재배정은 관리자만 수행합니다.
- Firestore에서 역할·담당 학생·문서 소유자를 검증하고, 문자열 크기·상태값·AI 검토 메타데이터를 서버 규칙에서도 제한합니다.
- 비활성화된 `users` 프로필은 커스텀 클레임이 남아 있어도 업무 문서에 접근할 수 없습니다.
- AI 초안 함수는 승인된 상담사인지와 해당 학생의 담당 상담사인지를 각각 검증합니다.
- AI 요청에는 학생 이름·학번·프로필·연락처를 포함하지 않으며, 메모 속 이메일·휴대전화·주민등록번호 형식은 전송 전에 마스킹합니다.
- AI가 만든 요약은 항목별 근거를 반환해야 하며 상담사가 근거 검토를 완료하지 않으면 저장할 수 없습니다.
- AI 검토는 요약·강점·고민·안내 네 항목을 각각 확인하고, 추가 확인이나 민감정보 경고가 있으면 별도로 승인해야 합니다.
- 신뢰 센터의 점수는 저장된 근거·검토·마스킹·만료 데이터를 가중 합산한 운영 지표이며 보안 인증을 의미하지 않습니다.
- AI 오류 로그에는 상담 메모, 프롬프트, 모델 응답 본문을 기록하지 않습니다.
- `consultationDrafts.expiresAt`을 기준으로 7일 TTL 정책을 활성화합니다. 코드의 만료 시각 기록만으로는 자동 삭제되지 않으므로 운영 인프라 설정을 별도로 확인합니다.
- 관리자 역할은 보안 사고·감사 목적의 예외 접근 권한을 포함합니다. 관리자 계정을 최소화하고 다중 인증과 Cloud Audit Logs 보존을 적용합니다.
- GitHub에서 비밀 검사, 푸시 보호, Dependabot 경고와 기본 브랜치 보호를 활성화합니다.
