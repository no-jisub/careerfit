# 커리어핏 개인정보·접근 권한 설계

## 1. 설계 목표

커리어핏의 상담 기록은 진로 고민, 개인 배경, 행동 계획처럼 민감도가 높은 정보를 포함할 수 있다. 따라서 편의 기능보다 다음 원칙을 우선한다.

1. **최소 수집**: 상담 운영에 필요하지 않은 직접 식별정보는 AI 처리 대상에서 제외한다.
2. **최소 권한**: 상담사는 배정된 학생의 문서만 조회하고 변경한다.
3. **목적 분리**: 내부 원문, 업무용 상담 기록, 학생 공개 요약을 별도 문서로 저장한다.
4. **사람의 최종 책임**: AI 결과는 초안이며, 근거를 상담사가 확인하기 전에는 최종 저장할 수 없다.
5. **투명성**: 학생에게 공개되는 요약에는 AI 보조 여부와 상담사 검토 여부를 표시한다.

## 2. 데이터 분류와 저장 위치

| 등급 | 데이터 | 저장 위치 | 학생 공개 |
|---|---|---|---|
| 제한 | 상담 담당자 원문 메모 | `consultationNotes` | 불가 |
| 제한 | 작성 중인 상담 양식 | `consultationDrafts` | 불가 |
| 내부 | 전체 상담 기록과 AI 근거 | `consultations` | 불가 |
| 공개 선택 | 상담사가 선택한 요약 필드 | `consultationSummaries` | 본인에게만 가능 |
| 업무 | 예약·후속 조치 | `appointments`, `followUps` | 본인 또는 담당자 |
| 신원 | 학생 프로필·배정 정보 | `students`, `users` | 본인 또는 권한자 |

원문 메모는 공개 요약 문서에 복사하지 않는다. 학생 화면은 `consultationSummaries`만 구독하며 `consultationNotes`와 `consultations`를 조회하지 않는다.

## 3. 역할별 접근 권한

| 기능 | 학생 | 담당 상담사 | 관리자 |
|---|---|---|---|
| 학생 프로필 | 본인 | 담당 학생 | 전체 |
| 내부 원문 메모 | 불가 | 담당 학생 | 보안 사고·감사 목적의 예외 접근 |
| 전체 상담 기록·AI 근거 | 불가 | 담당 학생 | 보안 사고·감사 목적의 예외 접근 |
| 학생 공개 요약 | 본인 공개분 | 담당 학생 | 전체 |
| 상담 임시 기록 | 불가 | 본인 작성분 | 불가 |
| 예약·후속 조치 | 본인 | 담당 학생 | 운영 관리 |
| 사용자 승인·학생 배정 | 불가 | 불가 | 가능 |

관리자 권한은 일상적인 상담 열람 권한으로 사용하지 않는다. 운영 환경에서는 관리자 계정 수를 최소화하고, Firebase/Google Cloud 감사 로그를 보존해 예외 접근을 사후 확인할 수 있어야 한다.

## 4. 실제 강제 지점

### Firestore

- `firestore.rules`가 인증 여부, 활성 계정, 역할, 담당 학생, 문서의 `counselorUid`와 `studentUid`를 검증한다.
- 상담사는 `where('counselorUid', '==', uid)` 또는 `where('ownerUid', '==', uid)` 조건으로 담당 자료만 구독한다.
- 사용자 승인과 학생 재배정은 `admin` 역할만 수행한다.
- 학생은 `published == true`인 본인 공개 요약만 읽는다.
- 임시 상담 기록은 작성 상담사 본인만 읽고 변경하며, 7일 뒤의 `expiresAt`을 기록한다.

### Storage

- 예약 첨부파일은 해당 예약의 학생 또는 담당 상담사만 읽고 업로드할 수 있다.
- 완료된 예약의 첨부파일은 학생이 삭제할 수 없다.
- 허용 파일 형식과 파일당 5MB 제한을 서버 규칙에서 검증한다.

### 애플리케이션

- 학생용 화면은 공개 요약 컬렉션만 사용한다.
- 상담사용 통합 검색은 현재 계정에 구독된 담당 학생·상담 기록만 대상으로 한다.
- 내부 원문과 공개 요약을 서로 다른 객체로 생성한다.

## 5. AI 상담 요약 처리 흐름

```text
상담사 입력
  → 담당 학생 권한을 Cloud Function에서 재검증
  → 학생 이름·학번·연락처를 요청 객체에서 제외
  → 메모 속 이메일·휴대전화·주민등록번호 형식 마스킹
  → AI 초안 + 항목별 근거 + 추가 확인 + 민감정보 경고 생성
  → 상담사가 내용과 근거를 검토
  → 검토 완료 시각·검토자·모델 정보를 함께 저장
  → 선택된 항목만 학생 공개 요약으로 분리
```

AI 오류 로그에는 상담 메모나 프롬프트를 기록하지 않는다. 오류 단계, 코드, 모델 응답 상태처럼 운영에 필요한 비식별 메타데이터만 남긴다.

## 6. 근거와 출처 표시

- AI는 `summary`, `strengths`, `concern`, `guidance`별 근거를 반환해야 한다.
- 근거가 없으면 반드시 `근거 부족`으로 표시한다.
- 상담사가 근거 검토를 완료하지 않으면 최종 저장을 차단한다.
- 내부 상담 기록에는 근거 목록과 추가 확인 항목을 보존한다.
- 학생 공개 요약에는 내부 근거 원문 대신 `AI 보조 초안을 상담사가 근거 확인한 요약`이라는 출처를 표시한다.

## 7. 신뢰 지표와 검토 이력

`/trust-center`는 저장된 운영 데이터에서 다음 지표를 계산한다. 수치는 홍보용 고정값이나 보안 인증 점수가 아니다.

- 근거 충족률 40%: 네 개 필수 항목 중 실제 근거가 있는 비율
- 사람 검토율 30%: 검토자와 검토 완료 시각이 기록된 AI 상담 비율
- 식별정보 마스킹 확인율 20%: 서버 마스킹 메타데이터가 기록된 비율
- 임시기록 만료 설정률 10%: 유효한 `expiresAt`이 있는 활성 초안 비율

검토 이력은 별도 감사 로그를 가장하지 않고, 상담 기록에 저장된 검토자·시각·근거 확인 필드에서 파생한다. 운영 환경의 실제 관리자 열람·규칙 거부·데이터 삭제 감사는 Google Cloud Audit Logs와 별도 보안 모니터링으로 보완해야 한다.

## 8. 보존·파기 정책

- 임시 상담 기록은 7일 후 삭제 대상이 되도록 `expiresAt`을 저장한다.
- 운영 배포 시 Firestore TTL 정책을 `consultationDrafts.expiresAt` 필드에 활성화해야 실제 자동 삭제가 이루어진다.
- 완료 상담 기록, 원문 메모, 첨부파일의 보존 기간은 대학의 기록물·개인정보 처리방침에 따라 확정해야 한다.
- 보존 기간이 끝난 데이터는 문서와 Storage 객체를 함께 삭제하고, 감사 가능한 파기 결과를 남겨야 한다.
